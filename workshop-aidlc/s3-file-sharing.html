Portable File-Sharing System

Build a complete web-based file sharing system that runs locally with open-source infrastructure

This workshop references a vision.md file. Download it first before starting.

Workshop Overview

Duration: ~4 hours | Project: Portable File-Sharing System

Build a web-based file sharing system with MinIO for S3-compatible object storage, ScyllaDB Alternator for metadata, a local API service for gateway-style routing, function-style backend handlers, and React for the frontend. Includes multiple user roles (Admin, Uploader, Reader, Viewer) with role-based permissions.

Portable stack: This workshop teaches the same cloud-native patterns without requiring AWS services. MinIO replaces S3, ScyllaDB Alternator replaces DynamoDB, local API routes replace API Gateway, function-style handlers replace Lambda functions, and Docker Compose replaces CloudFormation/SAM deployment.

Prerequisites

  • Basic React/JavaScript and Python knowledge
  • Access to an AI assistant (Kiro, Claude, Cursor, etc.)
  • Docker Desktop or Docker Engine with Docker Compose
  • Node.js, npm, and Python installed
  • Optional: MinIO Client (mc) for inspecting local buckets
  • Code editor

Construction Activity 7: Local Infrastructure + Deployment (~30 minutes)

The AI designs the portable local deployment architecture and generates Docker Compose configuration. This covers:

  • A MinIO service with persistent object storage and an initialized private bucket such as file-share-workshop
  • A ScyllaDB Alternator service with persistent metadata storage and table initialization scripts
  • A local API service that exposes gateway-style routes for auth, upload, download, sharing, and admin workflows
  • Function-style backend handler modules for file operations, permission checks, and metadata updates
  • A React frontend served from a static container or by the local API service
  • Docker Compose networking, named volumes, health checks, and environment variables

The AI will also handle local deployment using Docker Compose:

  • Create docker-compose.yml, .env.example, and service-specific Dockerfiles if needed

  • Run docker compose up --build and monitor container status
  • Verify local URLs for the frontend, API, MinIO console, and ScyllaDB Alternator endpoint
  • Use curl or an API test script to verify upload, list, share, and download flows
Expected local URLs: frontend at http://localhost:3000, API at http://localhost:8000, MinIO console at http://localhost:9001, and ScyllaDB Alternator at http://localhost:8001.

Task: Use the following steps to deploy the application code generated in the backend and react-app folders as a portable local developer stack. Do not require AWS services, AWS credentials, AWS CLI, SAM, CloudFormation, API Gateway, Lambda, or CloudFront.
Step 1: Create local infrastructure as code
Create a docker-compose.yml file that defines:
  - A MinIO service for S3-compatible object storage, with a persistent named volume and console exposed on localhost:9001
  - A ScyllaDB Alternator service for metadata and permissions, with a persistent named volume and endpoint exposed on localhost:8001
  - A backend API service exposed on localhost:8000 that provides gateway-style routes for auth, users, uploads, downloads, sharing, and admin workflows
  - A frontend service exposed on localhost:3000 that serves the built React application
  - Health checks, service dependencies, local network configuration, and named volumes
Create a .env.example file with all local endpoint URLs, table names, bucket names, access keys, and ports.
Step 2: Create initialization scripts
- Create repeatable scripts to initialize the MinIO bucket, access policy needed by the app, and ScyllaDB Alternator tables
- Make initialization idempotent so rerunning it does not corrupt existing local data
- Document how to reset local volumes when a clean environment is needed
Step 3: Build and run the local stack
- Build the React application
- Run docker compose up --build
- Monitor container status with docker compose ps
- Read container logs for the API, MinIO, ScyllaDB Alternator, and frontend services
Step 4: Verify the deployed application
- curl the backend health endpoint at http://localhost:8000
- Verify the frontend loads at http://localhost:3000
- Verify the MinIO console is reachable at http://localhost:9001
- Verify ScyllaDB Alternator is reachable at http://localhost:8001
- Run an integration test or script that uploads a real file to MinIO, writes metadata to ScyllaDB Alternator, lists the file, creates a share, and downloads the file
